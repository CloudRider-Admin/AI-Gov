# Govi — Performance & Template-Fidelity Report

_Authored 2026-05-26. Pairs with `docs/govi-architecture.md` (the as-built design) and `RECOMMENDATIONS.md` (the prior backlog)._

## TL;DR

- **Does Govi use the knowledge repo?** Partially. 57 of the 64 source files under `GovSecure Project-…/GovSecure Project/` are extracted at build time to JSON in `src/data/govsecureContent/`, and 22 GovSecure document types are built from that JSON. But the extractor's table content is dropped on its way to the LLM, the original DOCX/PDF templates are never opened at runtime, and `KnowledgeEntry` is not auto-seeded so the vector path cannot see the GovSecure library until an admin manually triggers `/api/knowledge/seed`.
- **Does it strictly follow template formats?** No. Section headings match the source because they are built from the extracted JSON. The actual content is regenerated by GPT-4o from short "guidance" strings; the **tables** that carry every policy's Purpose/Scope/Owner metadata are silently dropped on the way in, the branded DOCX/PDF exporters do not render tables at all, and there is no "verbatim" path for users who want the source policy with their org details substituted in.
- **How is it performing?** Single-shot orchestrators (intake / document / playbook = one OpenAI call each), single-shot advisor with Claude fallback. The "multi-agent" pipeline advertised in the README and code (`MultiAgentOrchestrator`, 4 weighted agents, consensus, synthesis) is **dead code** — exported but unreachable from any route. **Eval pass-rate is unknown** — every report on disk is either mock-mode (~0.03 s, deterministic synthetic outputs) or 100% errored on a real quota failure, so the CI gate is decorative.

### Coverage gaps

- **Three sources of truth for governance content**: `governanceKnowledge.ts` (980 lines, hand-curated NIST/EU AI Act/ISO), `nistPlaybook.json` (1496 lines), and the extracted `govsecureContent/`. They are not reconciled.
- **Two orphan duplicate files** (likely bot-generated import accidents): `src/data/src_src_data_nistPlaybook.json` (identical to `nistPlaybook.json`) and `src/lib/sanity/src_src_lib_sanity_fetch.ts` (identical to `fetch.ts`).
- **`KnowledgeEntry` table is not auto-seeded.** Migration files create the table; nothing populates it. RAG vector search is therefore dormant until `/api/knowledge/seed` is triggered manually. The vector branch in `rag.ts` short-circuits and falls through to keyword search of the static `governanceKnowledge.ts`.
- **Suite Map, Offering Guide, and Client-Ready packages** are extracted (`policies/suite-map.json`, `policies/suite-offering*.json`, `playbooks/ai-chef-*-2.json`) but not surfaced in any UI route.

## 2. Template fidelity

### What matches the source

- ✅ Section headings: `buildTemplates` reads `doc.sections` and emits the source `heading` verbatim for every GovSecure doc type.
- ✅ Document code: `buildDocumentCode` produces `GS-AIPS-AUP-01-{hash}` — same prefix pattern as the manifest's `GS-AIPS-ACCEPTAB-01`.
- ✅ Branded exporter visuals: title page (code, risk tier, date, review cycle), branded heading styles in the GovSecure palette, page footer with code + version + "Confidential — Internal Use Only", required license block as the last section.
- ✅ Canonical 4-tier risk model: `GOVSECURE_RISK_MODEL` mirrors the AI Chef toolkit Low/Moderate/High/Prohibited model with the same scoring dimensions.
- ✅ Few-shot voice anchor: `exemplarRetrieval.ts` injects 1-2 real prose passages from the source document into the prompt as a `REFERENCE EXEMPLARS` block.
- ✅ Anti-hallucination guardrails on intake (the `Insufficient information —` rule) and the citation validator's hard rejection of unknown `GS-…` codes / out-of-range article numbers.

### What does NOT match

**1. Tables are dropped on the way in AND on the way out (the biggest gap).**

The source AUP DOCX leads with a Purpose / Scope / Primary owner / Audience metadata table at the top of every major section. The extractor captures it (`acceptable-use.json` carries full `tables: [[…]]` arrays), but:

- `buildContext()` in `src/data/govsecurePolicies.ts` joins paragraphs and bullets only — it ignores `section.tables`. So the LLM never sees the metadata table that defines each section's identity.
- Neither branded exporter renders tables — `grep "table\|TableCell"` on `govSecureWordExporter.ts` and `govSecurePdfExporter.ts` returns nothing.

Users download a Govi-generated AUP that looks branded (font, code, footer) but is missing the metadata tables, role/responsibility tables, and cross-reference tables that the source policies use as load-bearing structural elements.

**2. Section content is regenerated, not faithfully transcribed.**

The orchestrator prompt says "_match their tone, structure, density, and terminology in your output. Do NOT copy them verbatim_." That is the right product call for license + customization, but it means the output is "in the style of GovSecure" rather than "an instance of the GovSecure template." There is no toggle to fall back to the verbatim source for sections the user wants unchanged.

**3. Section coverage is partial for some types.**

The Evidence Pack extraction returned only the preamble (1 section). `src/data/govsecurePolicies.ts:177-189` falls back to 12 hard-coded headings as a workaround. Any source whose extractor output is sparse drifts from the canonical structure.

**4. Output schema is normalized, source schema isn't.**

`governanceDocumentSchema` flattens each section into `{ heading, content: string, checklistItems?, required }`. The source documents have richer structure (numbered subsections, embedded tables, inline annotations, section-level framework cross-refs). All of that flattens to flowing paragraphs in the export.

**5. Generic governance docs have no template at all.**

`dpia`, `threat-model`, `model-card`, `monitoring-plan`, `risk-memo`, etc. (11 generic types) have hand-curated section templates in `documentTemplates.ts`, not derived from any source document. The fidelity story does not apply to these — they are internal authoring.

## 3. Performance

### Eval coverage is a mirage

Every eval report under `evals/reports/` says 100% pass except one that says 0% — and that 0% run is the only one that did real LLM calls (49.53 s, all 30 cases errored on `429 You exceeded your current quota`). The 13 passing runs each took 0.03–0.09 s — `EVAL_MOCK=1` deterministic synthetic outputs.

The CI gate (`.github/workflows/govi-eval.yml`, fails on >2 pp regression) is therefore a no-op — comparing a deterministic constant against itself. Nobody knows Govi's real pass rate.

### The "multi-agent" system is unreachable in production

`MultiAgentOrchestrator.orchestrateResponse` issues 5 OpenAI calls (1 primary + 3 supporting in parallel + 1 synthesis) with weighted-consensus voting (`risk-assessor 0.40, compliance-expert 0.30, policy-architect 0.15, implementation-advisor 0.15`) and a `contested` flag. The README and `docs/govi-architecture.md` describe this as the headline pipeline.

`grep multiAgentOrchestrator` outside tests: only `multiAgent.ts` itself. **No route calls it.** The three operational orchestrators (intake / document / playbook) each issue exactly one `callOpenAIJson` call. The advisor route uses a single GPT-4o call directly.

### Real-world latency hotspots

| Step | Wall time (typical) | Note |
|---|---|---|
| Auth + rate-limit + token-budget DB hits | 50–200 ms | Three sequential awaits; should be `Promise.all`. |
| Prompt-injection guard | 5–20 ms | In-memory regex. |
| `buildEnhancedRAGContext` | 200–800 ms | **Loads ALL active `KnowledgeEntry` rows** when vector search is unavailable. No `take`. Perf bomb at scale. |
| `dispatchOrchestrator` parallel reads (`getOrgContext`, `getIntakeProfile`) | 80–250 ms | Two Prisma reads against the same Conversation row. |
| GPT-4o advisor call | 3–8 s | `max_tokens=4000`, JSON mode. |
| Parse → validate → cite-check | 50–150 ms | In-memory. |
| Orchestrator dispatch GPT-4o call (when intent ≠ advisor) | +6–15 s | Another full round-trip, no streaming on the artifact. |
| Webhook dispatch (fire-and-forget) | varies | Fires after response; in serverless the worker may be reaped before POST completes. |
| Stream resume | depends on instance routing | `streamBuffer` is per-process; resume only works on the same warm instance. |

### Functional correctness gaps

- Streaming citation validation is post-hoc only. Once the SSE has shipped, a hallucinated `Article 250 of GDPR` cannot be stripped — only logged. Non-streaming path warns the user when `unverified > STRIP_THRESHOLD`.
- Webhook fan-out is fire-and-forget in a serverless handler. Vercel may terminate the function before `fetch` completes.
- Stream resume crosses instance boundaries badly. `streamBuffer` is process-local.
- JWT role caches until next sign-in. After Stripe upgrade, the user keeps the old role until they sign out and back in.
- `auditLog` is `console.log` only. The product's own value-prop audit trail does not persist.

## 5. What this branch fixes

- `src/lib/ai/rag.ts`: cap `KnowledgeEntry.findMany` at 200 rows ordered by `updatedAt desc`.
- `src/data/govsecurePolicies.ts`: `buildContext()` now serializes source `section.tables` as Markdown tables so the LLM sees the Purpose/Scope/Owner metadata it is supposed to mirror.
- `src/lib/ai/schemas.ts`: `governanceDocumentSchema.sections[]` gains an optional `tables: string[][][]` field.
- `src/lib/ai/multiAgent.ts`: `DocumentOrchestrator` prompt and JSON skeleton ask the LLM to emit a `tables` array per section when the source carried tables. `buildDocumentMarkdown` renders them.
- `src/lib/exporters/govSecureWordExporter.ts`: renders `section.tables` as branded `docx` `Table`/`TableRow`/`TableCell` blocks.
- `src/lib/exporters/govSecurePdfExporter.ts`: renders `section.tables` via a new `drawTable` helper using `jsPDF`.

Together these close the table-fidelity gap end-to-end: extracted JSON → LLM prompt → LLM output → markdown → DOCX → PDF.
